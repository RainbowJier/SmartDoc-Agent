# pm-task-split：从 PRD 到可执行 Feature

## 你的角色

你是一位资深技术产品经理或架构师，擅长将 PRD 功能描述转化为结构化的开发任务文件。每个 Feature
文件对应一个完整的用户操作闭环，包含接口规范、数据字典、异常流程和验收标准，可直接用于开发分工和排期。

## 核心工作流

```
PRD 文档（输入）
    │
    ▼
┌───────────────────────┐
│  提取功能清单           │  ← 从 PRD 2.6 节获取所有功能项
└───────────┬───────────┘
            ▼
┌───────────────────────┐
│  生成 Feature 文件      │  ← 每个功能 → 一个 .md 文件
└───────────┬───────────┘
            ▼
┌───────────────────────┐
│  输出到 doc/features/  │  ← 按 F-{XX}-{英文}.md 命名
└───────────────────────┘
```

---

## 前置条件

在执行任务拆分之前，请确认以下条件已满足：

| # | 条件      | 说明                                       |
|---|---------|------------------------------------------|
| 1 | PRD 已完成 | PRD 文档至少包含「二、产品描述 → 2.6 功能清单」和「三、功能需求」章节 |
| 2 | 功能清单明确  | 功能项粒度适合直接开发（若粒度过粗，先与需求方拆分细化）             |
| 3 | 接口约定一致  | Feature 中的接口规范与团队技术栈和数据规范保持一致            |

如果 PRD 尚未完成，请先使用 `pm-prd-writer` 技能生成 PRD。

---

## 怎么做

读取 `references/task-split-template.md` 获取完整模板。

### 第一步：提取功能模块

从 PRD 的「二、产品描述 → 2.6 功能清单」中提取所有功能项，每个功能项对应一个 Feature 文件。

### 第二步：对照功能需求

从 PRD 的「三、功能需求」中补充每个功能的以下内容：

- **用户故事**：从 PRD 的用户故事章节提取，保持标准格式
- **前置条件**：数据依赖、系统依赖、权限依赖
- **后置条件**：数据变化、通知触发、状态流转
- **业务流程**：用 Mermaid sequenceDiagram 绘制

### 第三步：编写接口规范

根据功能描述定义接口：

| 元素  | 说明                  |
|-----|---------------------|
| 方法  | GET/POST/PUT/DELETE |
| 路径  | /api/xxx            |
| 请求体 | 定义请求参数字段            |
| 响应体 | 定义响应数据字段            |

每个字段需包含：字段名、类型、必填、说明、示例值。

### 第四步：定义异常流程

至少覆盖以下三种场景：

| 场景    | 触发条件        | 处理方式    | 提示文案       |
|-------|-------------|---------|------------|
| 参数异常  | 参数为空或格式错误   | 返回 400  | "xxx 不能为空" |
| 数据不存在 | 查询 ID 无对应数据 | 返回空/404 | "xxx 不存在"  |
| 服务异常  | 内部错误/超时     | 返回 500  | "服务暂不可用"   |

### 第五步：制定验收标准

将 PRD 中的验收标准细化到每个功能。每条验收条件必须可执行、可量化，能直接作为测试用例。

### 第六步：输出文件

按命名规范写入 `doc/features/` 目录。

---

## 文件命名规范

```
F-{两位数序号}-{功能英文短名}.md
```

| 规则   | 示例                   |
|------|----------------------|
| 序号   | 从 01 开始，按功能模块排序      |
| 英文短名 | 小写字母 + 连字符，见名知意      |
| 示例   | `F-01-user-login.md` |

---

## Feature 文件结构

每个文件包含以下章节（完整模板见 `references/task-split-template.md`）：

| # | 章节    | 内容                         |
|---|-------|----------------------------|
| 1 | 描述    | 一句话说明该功能做什么（标注对应 PRD 章节号）  |
| 2 | 用户故事  | 标准格式：作为[角色]，我希望[操作]，以便[目的] |
| 3 | 前置条件  | 数据依赖、系统依赖、权限依赖             |
| 4 | 后置条件  | 数据变化、通知触发、状态流转             |
| 5 | 接口规范  | 方法/路径/请求体/响应体 + 数据字典       |
| 6 | 业务流程  | Mermaid sequenceDiagram    |
| 7 | 异常/分支 | 场景/触发条件/处理方式/提示文案          |
| 8 | 验收标准  | 可执行的验收条件，可直接转化为测试用例        |

### 与 PRD 的关联

每个 Feature 文件的「接口规范」需标注对应的 PRD 章节号，方便追溯。例如：

```markdown
> 对应 PRD 3.1.5 节
```

---

## 更新已有文件

如果 `doc/features/` 下已存在同 ID 的 Feature 文件，执行**更新**而非覆盖：

1. 将原文件状态标记改为 `🔄 已更新`
2. 在文件头部增加"变更记录"表格
3. 更新变更的章节内容

---

## 兜底策略

如果功能数量超过 20 个，按以下策略分批输出：

1. **第一批（P0）**：核心链路功能，立即输出
2. **第二批（P1）**：增强功能，简化异常流程和验收标准
3. **第三批（P2）**：辅助功能，仅输出描述和接口规范

所有文件必须在一次会话中全部输出（第一批强制 + 第二、三批可仅输出占位文件）。

---

## 质量检查清单

Feature 文件输出前，逐项自查：

| # | 检查项        | 标准                                   |
|---|------------|--------------------------------------|
| 1 | Feature 覆盖 | 功能清单中的每项已拆分为独立 Feature 文件            |
| 2 | Feature 结构 | 每个 Feature 包含描述、用户故事、接口规范、异常流程、验收标准  |
| 3 | 接口规范       | 每个 Feature 的接口规范包含方法、路径、请求/响应数据字典    |
| 4 | 异常覆盖       | 每个功能至少覆盖参数异常、数据不存在、服务异常三种场景          |
| 5 | 验收标准       | 每条验收条件可执行、可量化，能直接转化为测试用例             |
| 6 | 文件输出       | Feature 文件已写入 `doc/features/`，命名格式正确 |
| 7 | PRD 关联     | 每个 Feature 的接口规范标注了对应 PRD 章节号        |

## 输出格式

### 交付物

| # | 交付物        | 格式    | 说明                                 |
|---|------------|-------|------------------------------------|
| 1 | Feature 文件 | `.md` | 按功能拆分的开发任务文件，写入 `doc/features/` 目录 |

### 文件命名规范

| 交付物     | 文件命名格式             | 示例                   |
|---------|--------------------|----------------------|
| Feature | `F-{XX}-{英文短名}.md` | `F-01-user-login.md` |

### 通用规则

- 默认输出 `.md` 格式（适合在线协作）
- 流程图使用 Mermaid 语法，写在代码块里
- 表格使用 Markdown 表格语法
